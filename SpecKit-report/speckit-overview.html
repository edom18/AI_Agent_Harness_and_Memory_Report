SpecKit とは

SpecKit は、仕様駆動開発（Spec-Driven Development） を実現するためのツールキットです。GitHub の SpecKit プロジェクトからインストールされ、AI エージェント（Claude Code など）と連携して、ソフトウェア開発の計画・仕様作成・タスク生成・実装を体系的に進めることができます。

主な特徴

  • ドキュメントファースト: コードを書く前に、必ず仕様書と計画書を作成
  • 段階的な開発: 仕様 → 計画 → タスク → 実装という明確なフェーズ
  • AI エージェント対応: Claude Code、Gemini、Copilot など複数の AI ツールに対応
  • テンプレートベース: 一貫性のあるドキュメント構造を維持
  • バージョン管理統合: Git ブランチと連動した機能管理

ディレクトリ構造

.specify/
├── scripts/

│       ├── setup-plan.sh                # 計画ファイルのセットアップ

│   ├── spec-template.md                 # 機能仕様テンプレート

└── memory/
    └── constitution.md                  # プロジェクト憲章（開発原則）

主要コンポーネント

1. テンプレート (templates/)

SpecKit の核となるドキュメントテンプレート集です。

spec-template.md - 機能仕様テンプレート

  • 目的: 新機能の要件定義と受け入れ基準を記述
  • 内容:
    • ユーザーストーリー（優先度付き）
    • 受け入れ基準（Given-When-Then 形式）
    • 機能要件（FR-001、FR-002...）
    • 成功基準（測定可能な指標）
    • エッジケース

plan-template.md - 実装計画テンプレート

  • 目的: 技術的なアプローチと実装戦略を記述
  • 内容:
    • 技術スタック（言語、フレームワーク、DB）
    • プロジェクト構造（ディレクトリレイアウト）
    • Constitution チェック（開発原則への準拠確認）
    • 複雑性の追跡と正当化

tasks-template.md - タスクリストテンプレート

  • 目的: 実装を具体的なタスクに分解
  • 内容:
    • フェーズ分け（Setup → Foundational → User Stories → Polish）
    • タスク ID と説明
    • 並列実行可能なタスクのマーキング（[P]）
    • ユーザーストーリーごとのタスクグループ化
    • 依存関係の明示

checklist-template.md - チェックリストテンプレート

  • 目的: レビューや検証のためのカスタムチェックリスト
  • 内容:
    • カテゴリ別チェック項目
    • チェックボックス形式

agent-file-template.md - AI エージェント設定ファイルテンプレート

  • 目的: AI エージェント向けのプロジェクトコンテキスト提供
  • 内容:
    • アクティブな技術スタック
    • プロジェクト構造
    • コマンド（テスト、ビルド、lint など）
    • コーディング規約
    • 最近の変更履歴

2. スクリプト (scripts/bash/)

SpecKit ワークフローを自動化する Bash スクリプト集です。

common.sh - 共通関数ライブラリ

  • リポジトリルートの取得
  • 現在のブランチ/機能の特定
  • Git/非 Git 環境の判定
  • フィーチャーディレクトリのパス生成
  • ファイル/ディレクトリ存在確認

create-new-feature.sh - 新規機能の初期化

  • 機能:
    • 機能番号の自動採番（001、002...）
    • ブランチ名の生成とクリーニング
    • Git ブランチの作成（###-feature-name 形式）
    • specs/###-feature-name/ ディレクトリの作成
    • spec.md の初期化

setup-plan.sh - 計画フェーズの準備

  • 機能:
    • plan.md ファイルの作成
    • テンプレートのコピー
    • 環境変数の設定

check-prerequisites.sh - 前提条件チェック

  • 機能:
    • 機能ディレクトリの存在確認
    • 必須ファイル（plan.md、tasks.md）の確認
    • 利用可能なドキュメントのリストアップ
    • JSON/テキスト形式での出力対応

update-agent-context.sh - AI エージェントコンテキスト更新

  • 機能:
    • plan.md から技術情報を抽出
    • CLAUDE.md や GEMINI.md などのエージェントファイルを更新
    • プロジェクト構造、コマンド、技術スタックを自動反映
    • 複数の AI エージェントに対応（Claude、Gemini、Copilot、Cursor など）

3. Constitution（憲章） (memory/constitution.md)

プロジェクト全体の開発原則と品質基準を定義したドキュメントです。

主要な原則:

  • ドキュメントファースト開発: 実装前に必ず仕様書を作成
  • モジュラーアーキテクチャ: 疎結合で責任が明確なモジュール設計
  • テストによる品質保証: 自動テストで品質を検証
  • ユーザー体験重視: エンドユーザーを中心とした設計
  • コード品質基準: Lint、明確な関数シグネチャ、エラーハンドリング

バージョン管理: セマンティックバージョニングで変更を追跡

ワークフロー

SpecKit は以下の段階的なワークフローを推奨します。

1. 仕様作成 (/speckit.specify)
   ↓
2. 要件明確化 (/speckit.clarify) ※必要に応じて
   ↓
3. 実装計画 (/speckit.plan)
   ↓
4. タスク生成 (/speckit.tasks)
   ↓
5. 実装実行 (/speckit.implement)
   ↓
6. 分析とレビュー (/speckit.analyze)

各フェーズの詳細

1. 仕様作成 (/speckit.specify)

  • 自然言語の機能説明から spec.md を生成
  • ユーザーストーリー、機能要件、成功基準を定義
  • 独立してテスト可能なストーリーに分割（MVP アプローチ）

2. 要件明確化 (/speckit.clarify)

  • 仕様書の曖昧な部分を特定
  • 最大 5 つの質問で要件を明確化
  • 回答を仕様書に反映

3. 実装計画 (/speckit.plan)

  • 技術スタック選定
  • プロジェクト構造の決定
  • Constitution チェック（開発原則への準拠確認）
  • 設計ドキュメント生成（research.md、data-model.md、contracts/ など）

4. タスク生成 (/speckit.tasks)

  • ユーザーストーリーを実装可能なタスクに分解
  • フェーズとタスクID で構造化
  • 依存関係と並列実行可能性を明示
  • tasks.md ファイルを生成

5. 実装実行 (/speckit.implement)

  • tasks.md に基づいて実装を進行
  • チェックボックスでタスク完了を追跡
  • 各フェーズごとに検証ポイント設定

6. 分析とレビュー (/speckit.analyze)

  • 成果物の一貫性チェック
  • 品質分析レポート生成
  • 改善提案の提示

スキルコマンド

SpecKit は Claude Code のスキルシステムと統合されています。以下のコマンドが利用可能です。

コマンド 説明
/speckit.specify 機能仕様の作成・更新
/speckit.clarify 仕様の曖昧な部分を質問で明確化
/speckit.plan 実装計画の作成（技術選定、設計）
/speckit.tasks タスクリストの生成
/speckit.implement タスクの実装実行
/speckit.analyze 成果物の整合性分析
/speckit.checklist カスタムチェックリスト生成
/speckit.constitution プロジェクト憲章の作成・更新
/speckit.taskstoissues タスクを GitHub Issue に変換

Constitution（憲章）

Constitution は、プロジェクトの開発哲学と品質基準を定義する「契約書」です。

主要な3原則

  1. Documentation-First Development（ドキュメントファースト開発）

    • コード実装前に仕様書を作成
    • API 契約とデータモデルを設計段階で文書化
    • ドキュメントは成果物の一部

  2. Modular Architecture & Simplicity（モジュラーアーキテクチャとシンプルさ）

    • 単一責任の原則
    • 最小限の依存関係
    • YAGNI 原則（必要になるまで実装しない）
    • シンプルな解決策を優先

  3. Quality Through Testing（テストによる品質保証）

    • 公開インターフェースに対する契約テスト
    • 統合ポイントのテスト
    • 決定論的で独立したテスト

品質ゲート

  • 実装前ゲート: 仕様承認、技術アプローチ文書化、ユーザーシナリオ定義
  • マージ前ゲート: 全テスト合格、ドキュメント更新、コードレビュー完了

バージョン管理

Constitution はセマンティックバージョニングで管理されます。

  • MAJOR: 原則の削除や非互換な再定義
  • MINOR: 新原則の追加や実質的な拡張
  • PATCH: 明確化や非意味的な改善

まとめ

SpecKit は、仕様駆動開発を体系化し、AI エージェントと人間が協力して高品質なソフトウェアを構築するためのフレームワークです。テンプレート、スクリプト、憲章の3つの柱により、以下を実現します。

  • 一貫性: 統一されたドキュメント構造
  • 追跡可能性: 要件からタスクまでの明確なトレーサビリティ
  • 品質: 原則とゲートによる品質担保
  • 効率: 自動化されたワークフローと AI 連携

このガイドが、ProjectELU における SpecKit の活用に役立つことを願っています。
